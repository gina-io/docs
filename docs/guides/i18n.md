# Internationalisation (i18n)

Gina ships a small, dependency-free i18n core: per-bundle JSON catalogs, a
fallback chain that gracefully degrades from `fr_CA` through `fr` to a bundle
default, CLDR-correct plural forms via Node's built-in `Intl.PluralRules`, and
`{name}`-style parameter interpolation. No npm package to install — the runtime
primitives ship inside the framework.

This guide covers everything available today: the runtime API (`gna.t()`,
controller `self.t()`, swig + nunjucks `t` template filter), per-request
locale negotiation (`req.culture` from URL prefix / cookie / `Accept-Language`
/ settings default), the fallback chain, CLDR-correct plural forms, and the
`gina i18n:*` CLI for translator workflow (scan / add / export / import).

## Catalog files

Translation catalogs live per-bundle at `bundle/locales/<culture>.json`.
Filenames are either `<lang>.json` or `<lang>_<REGION>.json` — for example
`en.json`, `en_US.json`, `pt_BR.json`. The underscore form matches the
`GINA_CULTURE` env-var convention; hyphenated `en-US.json` is intentionally
not accepted — a `.json` file in `locales/` whose name does not match is skipped
with a warning at boot.

The `_validator` key is **reserved**: it holds per-culture overrides for the form
validator's built-in rule messages. See
[Localizing built-in error labels](/guides/forms-and-validation#localizing-built-in-error-labels).
Note those labels use `%`-tokens (`%s`, `%n`, `%l`), **not** the `{name}`
interpolation described below — they are read directly from the catalog rather than
through `t()`.

```json title="bundle/locales/en.json"
{
  "common": {
    "welcome": "Welcome!",
    "greeting": "Hello, {name}!",
    "items": {
      "one": "{count} item",
      "other": "{count} items"
    }
  },
  "errors": {
    "notFound": "Not found",
    "serverError": "Something went wrong"
  }
}
```

Catalogs are eager-loaded at bundle start into
`process.gina._i18nCatalogs[bundleName]`. Editing a catalog file requires a
bundle restart to pick up changes — same policy as `routing.json` and
`connectors.json`. Hot-reload is on the roadmap as a follow-up.

The bundle scaffold (`gina bundle:add`) ships a starter `en.json` with a few
common keys. Add additional cultures by creating new JSON files in the same
directory.

### Schema

Catalog files validate against the JSON Schema published at
`https://gina.io/schema/locales.json`. Add the `$schema` reference if your
editor needs it (most editors auto-detect from the path):

```json
{
  "$schema": "https://gina.io/schema/locales.json",
  "common": { "welcome": "Welcome!" }
}
```

## Translating in controller actions

Inside controller action methods, use `self.t(key, params)`:

```javascript
this.home = function(req, res, next) {
    var welcome = self.t('common.welcome');
    var greeting = self.t('common.greeting', { name: req.session.user.name });
    var cart = self.t('common.items', { count: req.session.cart.length });

    self.render({
        welcome:  welcome,
        greeting: greeting,
        cart:     cart
    });
};
```

`self.t()` auto-binds the current request's culture, resolved per-request by
the [locale negotiation](#locale-negotiation--reqculture) chain. The bundle
name is inferred from `local.options.conf.bundle` — no extra wiring needed.

### Overriding the auto-bound culture

For one-off lookups in a different language (e.g. rendering a confirmation
email in the user's preferred locale, even though the request was made in
the default culture), pass the culture as a third argument:

```javascript
var subject = self.t('email.confirmSubject', { name: user.name }, user.preferredCulture);
```

## Translating outside controller scope

For lib code, middleware, or any context without `self`, use the global
`gna.t(key, params, culture, options)`. Culture is required — there's no
per-request magic without explicit wiring:

```javascript
var msg = require('gina').t('common.welcome', {}, 'fr_CA', {
    bundleName: 'dashboard'
});
```

`bundleName` defaults to `process.env.GINA_BUNDLE` when omitted, so passing it
explicitly is mostly only needed in tests and tooling.

## Form validator labels

The [FormValidator](/guides/forms-and-validation)'s built-in rule error messages
(`Cannot be left empty`, …) localize from the **same catalog** as `t()`, under the
reserved `_validator.<rule>` key — on both the server and the client.

The browser bundle does not load `lib/i18n`, so it never reads a catalog itself.
Instead Gina resolves the negotiated culture's `_validator` subset on the server and
whispers it into the page as `gina.config.validatorLabels` (alongside the resolved
culture as `gina.config.culture`). Your app therefore needs no bootstrap code; a
`gina.validator.setErrorLabels(labels[, culture])` call remains available to override
the catalog **per key** at runtime.

See [Localizing built-in error labels](/guides/forms-and-validation#localizing-built-in-error-labels)
for the catalog shape, the precedence order, and the `%`-token rules.

## Translating in templates

Both swig and nunjucks engines ship a `t` filter with the same surface, so
templates port between engines unchanged:

```html
<h1>{{ "common.welcome" | t }}</h1>
<p>{{ "common.greeting" | t({ name: user.name }) }}</p>
<span>{{ "common.items"  | t({ count: 5 }) }}</span>
```

The filter is auto-registered on every request — no `setup.js` wiring
needed. Culture auto-binds from `req.culture` with a fallback to the
bundle's process default (`GINA_CULTURE`). Bundle name is read from the
per-request `options.conf.bundle` slot.

The filter is a thin pass-through to the same `lib/i18n` primitives the
controller helper uses, so fallback chain, plural rules, interpolation,
and missing-key behaviour all match `self.t()` byte-for-byte. The catalog
loaded for the bundle is the same single source of truth across all three
call surfaces (`gna.t`, `self.t`, template filter).

## Locale negotiation — `req.culture`

Every request gets a `req.culture` field set by the framework before
middlewares drain and the controller dispatches. Negotiation order
(highest priority first):

1. **URL prefix** — opt-in via `routing.json > "culturePrefix": true` per
   route. The route author writes a `:culture` URL param and the framework
   sets `req.culture` from it (validated against the bundle's loaded
   catalogs):

   ```json title="routing.json"
   {
     "dashboard": {
       "url": "/:culture/dashboard/:id",
       "method": "GET",
       "culturePrefix": true,
       "control": ":dashboardController::view"
     }
   }
   ```

   `GET /fr/dashboard/42` → `req.culture = "fr"`. If the captured prefix
   isn't in the bundle's available cultures, the negotiator falls
   through to the next source.

2. **Cookie** — the `gina_culture` cookie. Bundle authors typically set
   this cookie themselves in middleware after the user picks a language.

3. **`Accept-Language` header** — parsed (with `q=` quality values
   honoured) and best-matched against the bundle's loaded catalogs. The
   matcher walks each requested entry in order, trying the exact tag
   first (`fr_CA`) then the base language (`fr`).

4. **`settings.region.culture`** — bundle's default culture, set by
   `gina bundle:add` from the user's locale.

5. **`GINA_CULTURE`** — process default env var, set at framework init.

6. **`'en'`** — hardcoded last resort.

Cultures are stored on `req.culture` in underscore form (`en_US`, `fr`,
`pt_BR`). Convert to BCP-47 for `Intl.*` calls with
`req.culture.replace('_', '-')`.

The negotiator runs inside a try/catch so an i18n failure never blocks
routing — worst case `req.culture` falls back to `GINA_CULTURE` or
`'en'`.

### Bundle-side overrides

Middleware can override the auto-negotiated `req.culture` by setting it
before the controller dispatch. Useful for sticky-locale flows where a
user has explicitly picked a language that should outlive the cookie:

```javascript
// bundle/middlewares/sticky-culture.js
module.exports = function(req, res, next) {
    var stored = req.session && req.session.userCulture;
    if (stored) {
        req.culture = stored;
    }
    next();
};
```

The controller helper `self.t()`, the template `t` filter, and the
existing region-data lookup all consume `req.culture` and stay in sync.

## Fallback chain

When a key is missing from the requested culture's catalog, lookup walks an
ordered fallback chain:

1. **Requested culture** — e.g. `fr_CA`
2. **Base language** — `fr`
3. **Custom chain** — from `settings.json > i18n.fallbackChain` (optional)
4. **Bundle default culture** — from `settings.json > region.culture`
5. **Process default** — `GINA_CULTURE` env var
6. **Last resort** — `'en'`

Each step is per-key, so a partial translation (50% of keys translated to
`fr_CA`, the rest still in `en`) cascades naturally without breaking the page.
Use `gina i18n:scan` to report coverage gaps.

If no catalog has the key, `t()` returns the key verbatim. In dev mode
(`NODE_ENV_IS_DEV=true`), missing keys are prefixed with the marker from
`settings.json > i18n.devMissingKey` (default `[MISSING]`) and a one-time
`console.warn` per `(bundle, key)` fires to help you spot gaps.

## Plural forms

Pluralisation uses Node's built-in `Intl.PluralRules` (CLDR-backed). Catalog
authors provide an object whose keys are CLDR plural categories — `zero`,
`one`, `two`, `few`, `many`, `other`. Different languages use different
subsets; English uses `one` + `other`, Russian uses `one`/`few`/`many`/`other`,
Arabic uses all six. Missing categories fall back to `other`.

```json title="bundle/locales/en.json"
{
  "common": {
    "items": {
      "one":   "{count} item",
      "other": "{count} items"
    }
  }
}
```

```json title="bundle/locales/ru.json"
{
  "common": {
    "items": {
      "one":   "{count} штука",
      "few":   "{count} штуки",
      "many":  "{count} штук",
      "other": "{count} штуки"
    }
  }
}
```

```javascript
self.t('common.items', { count: 1 });   // → "1 item"   (en) | "1 штука"  (ru)
self.t('common.items', { count: 5 });   // → "5 items"  (en) | "5 штук"   (ru)
self.t('common.items', { count: 21 });  // → "21 items" (en) | "21 штука" (ru, "one" form)
```

Plural lookup triggers when **both** conditions hold: the caller passes a
numeric `count` parameter, AND the resolved value is an object whose keys are
all CLDR plural categories. If either condition is false, the value is treated
as a string (or as a deeper nested category) and returned verbatim.

## Parameter interpolation

`{name}`-style placeholders are replaced with the matching value from the
`params` object:

```javascript
self.t('common.greeting', { name: 'Ada' });   // → "Hello, Ada!"
self.t('cart.summary',    { count: 3, total: '$42.00' });
```

Missing parameters leave the placeholder as a literal — `Hello, {name}!` is
returned when `params.name` is undefined. Use `{{name}}` to render a literal
brace pair (single de-quote pass): `Literal {{x}}` renders as `Literal {x}`.

For complex formatting (dates, numbers, currency), pass the formatted string
in via `params` and let `Intl.DateTimeFormat` / `Intl.NumberFormat` do the
work at the call site.

## Bundle settings — `i18n.*` block

Runtime knobs live under `settings.json > i18n`:

```json
{
  "i18n": {
    "cookieName": "gina_culture",
    "devMissingKey": "[MISSING]",
    "fallbackChain": null,
    "cultures": null
  }
}
```

| Key | Type | Default | Purpose |
| --- | --- | --- | --- |
| `cookieName` | string \| null | `"gina_culture"` | Cookie name for locale persistence. **Not currently consumed** — the locale negotiation reads the fixed cookie name `gina_culture` regardless of this setting. |
| `devMissingKey` | string \| null | `"[MISSING]"` | Dev-mode prefix for missing keys. Only fires when `NODE_ENV_IS_DEV=true`. `null` disables the marker. |
| `fallbackChain` | string[] \| null | `null` | Optional override of the default fallback chain. Inserted between the base-language and bundle-default lookups. |
| `cultures` | string[] \| null | `null` | Optional explicit allowlist for URL-prefix locale matching. `null` derives from the bundle's loaded catalogs. |

The bundle's default culture lives at `settings.json > region.culture` (the
existing slot — not duplicated under `i18n`):

```json title="config/settings.json"
{
  "region": {
    "culture": "fr_FR"
  }
}
```

## Backwards compatibility — `__()` legacy alias

The legacy `__(str)` placeholder (defined in `framework/v*/helpers/text.js`)
is rewired as a one-arg alias of `t()`. Existing callers continue to work:

```javascript
__('common.welcome');   // → catalog lookup if loaded, else "common.welcome" verbatim
```

Since `__()` takes no culture argument, lookup only succeeds when the bundle
catalog happens to have an entry for the bundle's process-default culture.
For real translation, prefer `gna.t()` or `self.t()` with explicit culture.

## Translator workflow and ICU

The runtime primitives, the controller helper, the swig + nunjucks `t`
filter, and per-request locale negotiation (`req.culture`) are covered
above. The framework also ships:

- **CLI** (*New in 0.3.11*) — `gina i18n:scan` (untranslated-key coverage
  report), `gina i18n:add <culture>` (seed a new culture from the default),
  `gina i18n:export` / `gina i18n:import` (`.po` / `.csv` / `.json`
  translator round-trip). Run `gina help i18n` for the full flag reference.
- **ICU MessageFormat** (*New in 0.3.11*) — `{count, plural, one {# item} other {# items}}`,
  `{gender, select, female {…} male {…} other {…}}`, and nested combinators.
  Covers gender-marked grammars (Romance, Slavic) and select expressions that
  the v1 `{name}` + plural-forms shape can't express. Opt-in via the
  `t.icu(key, params, culture)` variant; the existing `t()` shape is unchanged.
