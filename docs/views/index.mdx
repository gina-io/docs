Gina renders HTML through a **pluggable template engine**. Controllers call `self.render(data)` and Gina dispatches the call to whichever engine is configured for the bundle. The engine is chosen per bundle via `config/settings.json > render.engine`, and every engine receives the same `data.page` contract — the choice is a syntax decision, not an architectural one.

## Available engines

| Engine | Package | Syntax | Where it lives | When to pick |
|---|---|---|---|---|
| **[Swig](./swig)** (default) | `@rhinostone/swig` | Jinja2/Django-inspired (`{{ }}`, `{% %}`) | Framework dependency, ships with every bundle | Default for new bundles; keeps the framework's historical syntax |
| **[Swig-Twig](./swig/twig)** | `@rhinostone/swig-twig` | Twig (`{{ }}`, `{% %}`, stricter) | Drop-in swig replacement via `swig.package` | You have an existing Twig codebase, or your team writes Twig |
| **[Nunjucks](./nunjucks)** | `nunjucks` (project-side) | Jinja2 (`{{ }}`, `{% %}`, async-friendly) | **Project-only** — never a framework dep | You want native Jinja2, async loaders, or nunjucks-specific extensions |

Swig is the safe default. Swig-Twig swaps the parser on the same engine, so migration is near-zero-cost. Nunjucks is a full parallel engine — opt in only when you need it, because it fails loud if the project hasn't `npm install`ed it.

At this surface level the three engines look almost identical — that's by design. The differences surface in custom tags, filter registration, inheritance semantics, and how missing variables render. Each engine's section documents its own surface.

## Picking an engine

### Swig (default)

Nothing to configure. New bundles render via swig out of the box. Templates go in `<bundle>/views/`, `self.render(data)` resolves the route's `param.file` and renders it with `data.page` in scope.

[Read the Swig section →](./swig)

### Swig-Twig

Swap the package via `config/settings.json`:

```json
{
  "swig": {
    "useProject": true,
    "package": "@rhinostone/swig-twig"
  }
}
```

The framework picks up your project's `@rhinostone/swig-twig` from `node_modules/` at bundle start. Templates must be written in Twig syntax — see the [Swig-Twig section](./swig/twig) for the migration notes and parity table.

### Nunjucks

Install nunjucks in the **project** (not the framework) and flip `render.engine`:

```bash
cd <project>
npm install nunjucks
```

```json
{
  "render": {
    "engine": "nunjucks"
  }
}
```

The framework never declares nunjucks as a dependency — that's the core safety contract. Bundles that don't opt in pay zero cost and never have nunjucks in their runtime. Bundles that opt in without installing the package fail loud at startup with `NUNJUCKS_NOT_INSTALLED`.

[Read the Nunjucks section →](./nunjucks)

## Where the dispatch happens

`core/controller/controller.js :: this.render()` reads `conf.render.engine` from the bundle's resolved settings and routes the call to one of:

- `controller.render-swig.js` (engine `"swig"`, the default — also covers `swig-twig`)
- `controller.render-nunjucks.js` (engine `"nunjucks"`, loaded only when opt-in is set)

The same `data.page` shape is passed in either direction, so controller code is engine-agnostic. A bundle can switch engines by changing a single settings key — no controller changes needed.
