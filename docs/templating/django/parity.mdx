# Django Parity

This page lists exactly what `@rhinostone/swig-django` accepts â€” operators, variable resolution, tags, and filters â€” cross-referenced against Django Template Language (DTL) and the native `@rhinostone/swig` frontend.

The matrices below are ground truth: each row maps to a symbol actually exported by the shipped source, and every filter was cross-checked against Django 5.2's `defaultfilters.py`. Outputs match the reference except where the JavaScript runtime forces a divergence â€” those are flagged here and detailed under [Behavioural differences](#behavioural-differences). For features that throw at parse time, need the framework runtime, or are deferred, see [Non-Goals](./non-goals).

## A lenient superset of DTL

The expression grammar is a **lenient superset** of Django's. It keeps the inherited expression machinery (parentheses, arithmetic, comparisons) and adds the Django pieces (colon-filters, `True`/`False`/`None`, `is`/`is not` identity). Every valid Django template renders; the engine *also* accepts a handful of non-Django expressions â€” `{{ a + b }}`, `{{ (a or b) and c }}` â€” that real DTL never contains (Django has no arithmetic operators and no parentheses inside `{% if %}`). That extra acceptance is harmless: a valid Django template never uses those forms, so they cost nothing in fidelity. The tag and filter catalogs, by contrast, are **subsets** â€” the framework-coupled tags need a Django runtime swig-core does not provide (see [Non-Goals](./non-goals)).

## Operators

| Operator | Shape | Semantics | Lowering / notes |
| --- | --- | --- | --- |
| `\|name:arg` | colon-filter | Apply a filter with one positional argument â€” Django's filter-argument syntax. | The backend emits positional filter args; `\|date:"Y-m-d"` and `\|truncatewords:2` lower the same way. |
| `==` `!=` `<` `>` `<=` `>=` | comparison | Standard comparisons. | |
| `and` `or` `not` | logic | Short-circuit boolean logic. | |
| `in` | `x in seq` | Membership test. | `not in` does **not** parse â€” use `not (x in seq)`. See [Non-Goals](./non-goals#not-in-operator). |
| `is` / `is not` | `x is None` | Identity comparison (`===` / `!==`), **not** a Jinja2-style test. | `is None` / `is not None` route through the existence check so a missing value is distinguished from a real one. See [The `is` operator](#the-is-operator). |
| `True` `False` `None` | literals | Python-style boolean and null literals. | Lower to JavaScript `true` / `false` / `null`. |
| `( )` | grouping | Parenthesised sub-expressions (lenient â€” real DTL `{% if %}` has no parens). | |
| `+` `-` `*` `/` `%` | arithmetic | Accepted (lenient â€” real DTL has no arithmetic operators). `+` concatenates when either side is a string (JavaScript coercion). | Use the `add` filter for Django-faithful addition. |

Django's auto-calling of callables (no `()` needed â€” `{{ user.get_full_name }}`, not `{{ user.get_full_name() }}`) is part of [variable resolution](#variable-resolution), not an operator.

## The `is` operator {#the-is-operator}

In Django, `is` is an **identity comparison** â€” `{% if x is None %}`, `{% if flag is True %}` â€” not Jinja2's predicate-test mechanism (`x is defined`, `x is odd`, â€¦). `@rhinostone/swig-django` matches Django:

| Shape | Semantics |
| --- | --- |
| `x is None` | `true` if `x` is absent or `null` (routed through the existence check, so a missing context slot is distinguished from a real value). |
| `x is not None` | `true` if `x` is present. |
| `x is True` / `x is False` | Strict equality against the boolean. |
| `x is y` | Strict identity (`x === y`). Distinct object references are **not** identical (`{} is {}` â†’ false). |

Jinja2-style tests are **not** registered. `{% if x is defined %}` does **not** test definedness â€” it compiles to `x === defined`, comparing `x` against an (undefined) variable named `defined`, which is almost never what you want. There is no `is odd` / `is iterable` / `is mapping`. This is faithful to Django, which has no test system; if you are porting from Jinja2, see [Non-Goals â†’ Jinja2-style `is` tests](./non-goals#jinja2-style-is-tests).

## Variable resolution {#variable-resolution}

`{{ a.b.c }}` resolves each segment with Django's lookup algorithm, via the shared `swig-core` resolver (opt-in for this frontend; native/swig-twig/swig-jinja2 keep the plain dot-path):

1. **Dictionary lookup** â€” `obj["seg"]`.
2. **Attribute / method lookup** â€” `obj.seg`; if the result is a **function**, it is **auto-called** with no arguments (bound to its receiver) and the return value is used.
3. **Numeric index** â€” `{{ items.0 }}` resolves `items[0]` (JavaScript indexes arrays and strings by numeric string key, so one access covers all four cases).

```django
{{ user.get_full_name }}   {# calls user.get_full_name() â€” no parens #}
{{ items.0 }}              {# first element #}
{{ matrix.1.2 }}           {# nested index #}
```

Auto-call opt-outs match Django:

| Function flag | Effect |
| --- | --- |
| `fn.alters_data === true` | Not called â€” resolves to `undefined` (renders `""`). Mirrors Django's `alters_data`. |
| `fn.do_not_call_in_templates === true` | Returned **uncalled** (the function value itself). Mirrors Django's `do_not_call_in_templates`. |

A call that throws resolves to `undefined` (renders `""`), mirroring Django's `string_if_invalid` default. Every segment is still checked against the CVE-2023-25345 `_dangerousProps` blocklist at parse time, and the resolver re-checks at runtime as defence in depth â€” `{{ a.__proto__ }}` throws.

## Tags

`@rhinostone/swig-django` ships **12 built-in tag families** (`elif`, `else`, and `empty` are branch markers consumed by `if`/`for`, not standalone tags). Each is registered in `packages/swig-django/lib/tags/index.js`.

| Tag | Ends | Notes |
| --- | --- | --- |
| `if` / `elif` / `else` / `endif` | Yes | Full multi-branch conditionals, including `elif` chains and `and`/`or`/`not`/`in`/`is` conditions. |
| `for` / `empty` / `endfor` | Yes | `forloop.counter`/`counter0`/`revcounter`/`revcounter0`/`first`/`last`/`parentloop`, empty-fallback `{% empty %}`, two-variable unpack `{% for k, v in mapping %}`, and dict pseudo-methods `{% for k, v in d.items %}` / `d.keys` / `d.values`. The `reversed` modifier is [deferred](./non-goals#deferred-tag-forms). Loop variables containing `.` are rejected (CVE-2023-25345 guard). |
| `block` / `endblock` | Yes | Named block for template inheritance. `{% endblock name %}` accepted. `{{ block.super }}` is [deferred](./non-goals#deferred-tag-forms). |
| `extends` | No | Static (`{% extends "base.django" %}`) and dynamic (`{% extends layout_var %}`) â€” the dynamic form resolves on the [async render path](../swig/loaders#async-loaders); the synchronous path requires a string literal. See [Non-Goals](./non-goals#dynamic-extends). |
| `include` | No | `{% include "partial.django" %}` with Django's space-separated `with k=v` context and trailing `only` for isolation. No `ignore missing` (Django raises) â€” see [Behavioural differences](#behavioural-differences). |
| `with` / `endwith` | Yes | Both forms: modern `{% with a=1 b=2 %}` (space-separated pairs) and legacy `{% with expr as name %}`. |
| `autoescape` / `endautoescape` | Yes | `{% autoescape on %}` / `{% autoescape off %}` â€” Django's `on`/`off` spelling, not boolean literals. |
| `spaceless` / `endspaceless` | Yes | Collapses whitespace **between** HTML tags (`>\s+<` â†’ `><`) within the block only. Narrower than native swig's `spaceless` â€” see [Behavioural differences](#behavioural-differences). |
| `comment` / `endcomment` | Yes | Discards the body unparsed (broken tag syntax inside is safe). Optional `{% comment "note" %}` label accepted and ignored. |
| `verbatim` / `endverbatim` | Yes | Verbatim content; variable, tag, and comment syntax pass through untouched. |
| `cycle` | No | `{% cycle "a" "b" "c" %}` â€” cycles through values across loop iterations, with independent state per occurrence. The `as name` / named-reuse / `silent` forms are [deferred](./non-goals#deferred-tag-forms). |
| `firstof` | No | `{% firstof a b c %}` â€” outputs the first truthy argument (or `""`). The `as var` form is [deferred](./non-goals#deferred-tag-forms). |

There are **no** `macro` / `import` / `from` tags â€” Django has none either. Compose with `{% include %}`, `{% block %}`, and `{% extends %}`. The framework-infrastructure tags (`{% url %}`, `{% load %}`, `{% static %}`, `{% csrf_token %}`, `{% trans %}`, `{% blocktrans %}`) throw `Unexpected tag "<name>"` at parse time â€” see [Non-Goals â†’ Framework infrastructure](./non-goals#framework-infrastructure).

### Safe filters

These bypass autoescape (the autoescape `e` tail is not appended) â€” exactly six: **`safe`**, **`escapejs`**, **`force_escape`**, **`linebreaks`**, **`linebreaksbr`**, **`linenumbers`**. The three `linebreak*` / `linenumbers` filters escape their text content *before* wrapping it in markup, so marking them safe is XSS-correct. `escape` / `e` are **not** `.safe` (idempotency, not a flag, prevents double-escaping), and `floatformat` is deliberately **not** `.safe` â€” its error path returns the raw input, so a blanket safe flag would be an XSS hole (its numeric output carries no HTML, so the redundant escape is a harmless no-op). Filters that emit already-safe HTML you register yourself must set `.safe = true` on the function.

This is **not** the same as Django's `is_safe=True`: in Django, `is_safe` only marks that a filter *preserves* an already-safe input; only filters Django wraps in `mark_safe()` should carry swig's `.safe = true`.

### Registering your own

```js
django.setFilter('exclaim', function (input) { return String(input) + '!'; });
django.render('{{ msg|exclaim }}', { locals: { msg: 'hi' } });
// â†’ "hi!"
```

```js
function monospace(input) { return '<code>' + input + '</code>'; }
monospace.safe = true;
django.setFilter('monospace', monospace);
```

See [Extending Swig](../swig/extending) for the full filter/tag extension contract â€” it is identical across frontends.

## Behavioural differences

These behaviours are **supported** but diverge from Django because the JavaScript runtime (or the shared swig-core engine) differs from CPython. None of them throw â€” they just behave differently:

| Area | Django | `@rhinostone/swig-django` |
| --- | --- | --- |
| `{% for x in mapping %}` (single var) | Iterates the dict's **keys**. | Iterates the dict's **values** (swig object iteration). Use `{% for k in d.keys %}` for keys, `{% for k, v in d.items %}` for pairs. |
| `forloop.length` | No such field. | Present (swig exposes it) â€” harmless extra acceptance. |
| `default` on `""` | Substitutes (the value is falsy). | Same â€” substitutes. A missing variable also arrives as `""` after engine coercion, so `default` fires on it too. |
| `{{ True }}` / `{{ False }}` | Render `"True"` / `"False"`. | Render `"true"` / `"false"` (JavaScript booleans). Rarely output directly; in conditions they behave correctly. |
| `{{ None }}` | Renders `"None"`. | Renders `""` (engine null-coercion). |
| `{% include %}` `ignore missing` | Not a thing â€” a missing include raises. | Also raises (no `ignore missing` clause); Django-faithful. |
| `spaceless` | Strips whitespace between tags across the whole rendered block. | Same intent, region-isolated: only the block's own `>\s+<` runs collapse â€” narrower than native swig's whole-`_output` mutation. |
| `floatformat` | `mark_safe`s the success path only. | Not marked safe (the error path returns raw input); numeric output carries no HTML, so output is identical. |
| Object stringification | Python `str()` / `repr()`. | JavaScript `String(x)` â€” objects without a custom `toString` render `"[object Object]"`; `null` / `undefined` render `""`. |
| `date` / `time` | `strftime`-style format codes, locale-aware. | PHP-style codes via the shared `swig-core` formatter; no locale plumbing. Pass an explicit format string. |

See [Migrating from Django](./migration) for the porting view of these.

## Autoescape

Autoescape behaves as in native swig â€” `true` by default, applies the `e` filter last, honours `.safe = true` on filter exports. The built-in safe filters (the six listed above) are excluded. `{% autoescape off %}â€¦{% endautoescape %}` disables it for a region using Django's `on`/`off` spelling.

See [Security â€” what the guards protect against](../swig/security#what-the-guards-protect-against) for the full CVE-2023-25345 guard matrix â€” it applies uniformly across frontends because the guards live in `swig-core`'s shared backend.
