# Installation

Gina is distributed on npm and installs on either **Node.js** (with `npm`) or the **[Bun](https://bun.sh)** runtime (with `bun`). Because it acts as both a framework and a deployment environment, it is recommended to install it **globally**. A global install gives you the `gina` CLI, which manages projects, bundles, ports, and the framework socket server from any directory on your machine.

## Method 1 — Custom prefix (recommended)

This installs Gina in your home directory, avoiding the need for `sudo` or root access.

<Tabs groupId="runtime">
<TabItem value="npm" label="npm" default>

```bash
npm install -g gina@latest --prefix=~/.npm-global
```

Add `~/.npm-global/bin` to your `PATH` if it is not already there:

```bash
export PATH="$HOME/.npm-global/bin:$PATH"
```

Add that line to your shell profile (`.zshrc`, `.bashrc`, etc.) to make it permanent.

:::note npm 12+
npm 12 blocks install scripts by default, and Gina's post-install bootstraps `~/.gina` and the framework dependencies. Add `--allow-scripts=gina` to the install command, or allow it once for all global installs with `npm config set allow-scripts=gina --location=user`. Not needed on npm ≤ 11.
:::

</TabItem>
<TabItem value="bun" label="Bun">

On the [Bun](https://bun.sh) runtime (Bun `>= 1.2`):

```bash
bun add -g gina
```

Bun installs the `gina` CLI to `~/.bun/bin` — no custom prefix or `sudo` needed. If that directory is not already on your `PATH`, add it:

```bash
export PATH="$HOME/.bun/bin:$PATH"
```

Bun skips dependency install scripts by default, but Gina needs no extra setup — it self-bootstraps on first run, so there is no `trustedDependencies` entry to add.

</TabItem>
</Tabs>

### Verify the installation

```bash
gina version
```

`gina version` is a shortcut for `gina framework:version`.

## Method 2 — Classical global install

<Tabs groupId="runtime">
<TabItem value="npm" label="npm" default>

```bash
npm install -g gina@latest
```

(On npm 12+, add `--allow-scripts=gina` — see the note in the user-local install section above.)

If you encounter a permission error like `EACCES: permission denied, mkdir '/usr/local/lib/node_modules/gina'`, fix permissions first:

```bash
sudo chown -R $USER $(npm config get prefix --quiet)/lib/node_modules
```

Or install with `--unsafe-perm` (not recommended for production):

```bash
npm install -g --unsafe-perm gina@latest
```

> **Linux & macOS**: the use of `sudo npm install` is discouraged. Prefer Method 1 or fix permissions as shown above.

</TabItem>
<TabItem value="bun" label="Bun">

```bash
bun add -g gina
```

Bun's global install goes to `~/.bun/bin` in your home directory, so there are no `EACCES` permission errors and no `sudo` is required.

</TabItem>
</Tabs>

## Method 3 — Local install (single-project use)

If you only need Gina for a single project and do not want a global install:

<Tabs groupId="runtime">
<TabItem value="npm" label="npm" default>

```bash
npm install gina@latest
```

</TabItem>
<TabItem value="bun" label="Bun">

```bash
bun add gina
```

</TabItem>
</Tabs>

With this method, run the CLI using `./gina` from your project root instead of `gina`.

## Factory reset

If your preferences are broken or you want a clean slate, **`gina framework:reset`** (shorthand `gina reset`) clears `~/.gina` — your settings, project registry, env config, and port allocations — so it is rebuilt from defaults on the next command. Your project source files are never touched.

```bash
gina reset
```

It refuses while the framework daemon or any bundle is still running (so a reset can't orphan them) — stop them first with `gina stop`, or pass `--force`.

<Tabs groupId="runtime">
<TabItem value="npm" label="npm" default>

`gina reset` works on npm. On npm ≤ 11 you can also reset as part of a reinstall — this repairs the installed package **and** clears the preferences in one step:

```bash
npm install -g gina@latest --reset
```

npm 12 rejects unknown flags like `--reset` outright (`EUNKNOWNCONFIG`) — there, run `gina reset` first, then reinstall with `npm install -g gina@latest --allow-scripts=gina`.

</TabItem>
<TabItem value="bun" label="Bun">

`gina reset` is the way to factory-reset on Bun. The npm-style `bun add -g gina --reset` does **not** reset anything — Bun skips install scripts, so the `--reset` flag never runs:

```bash
gina reset
```

</TabItem>
</Tabs>

### When the `gina` command itself won't run

A reset command can't repair a broken **installation** — you can't run a broken program to fix itself. Recover in two steps: reinstall the package, then clear the preferences.

<Tabs groupId="runtime">
<TabItem value="npm" label="npm" default>

```bash
npm install -g gina@latest --reset
```

</TabItem>
<TabItem value="bun" label="Bun">

```bash
bun add -g gina      # reinstall the package
rm -rf ~/.gina       # clear preferences — rebuilt on the next command
```

</TabItem>
</Tabs>

`rm -rf ~/.gina` is the runtime-independent fallback that always works, no matter how broken things are — the next `gina` command re-creates `~/.gina` from defaults.

## Uninstalling

<Tabs groupId="runtime">
<TabItem value="npm" label="npm" default>

```bash
npm uninstall -g gina
```

If you installed with a custom prefix (`--prefix=~/.npm-global`):

```bash
npm uninstall -g gina --prefix=~/.npm-global
```

</TabItem>
<TabItem value="bun" label="Bun">

```bash
bun remove -g gina
```

</TabItem>
</Tabs>

Uninstalling removes the CLI but leaves your preferences in `~/.gina`. To remove those too:

```bash
rm -rf ~/.gina
```
